System architecture

Barkhaus is a static-frontend + Backend-as-a-Service architecture: all HTML/JS/CSS is served by GitHub Pages under the custom domain barkhaus.ph; every dynamic capability (database, auth, storage, realtime, serverless functions, scheduled jobs) is Supabase; payments and email are external SaaS reached from edge functions.

  • No customer accounts: the public flow runs on the anon key with row-level security limiting what it can read/write; identity is per-booking.
  • Admin is a separate SPA compiled from admin-src/ into /admin/; it authenticates real users (Google) and operates on the same database under authenticated RLS policies.
  • Money paths never trust the client: booking confirmation and all post-payment writes happen inside the webhook edge function using the service-role key.

4.2 Admin SPA

Backend — edge functions & scheduled jobs

All functions are Deno/TypeScript on Supabase, using the service-role key (RLS bypass) for writes. CORS open; secrets via function env (MAYA_PUBLIC_KEY, MAYA_SECRET_KEY, retained PayMongo keys, and RESEND_API_KEY).

FunctionCallerBehavior
create-maya-checkoutbooking.js (dormant Maya option)Validates the payload → creates the pending booking and one-hour hold → calls Maya Checkout with the public API key → stores the provider checkout ID → returns Maya's hosted checkout_url.
create-paymentbooking.js (retained PayMongo option)Retained PayMongo Checkout Session creator. It is hidden while PAYMENT_GATEWAY_PROVIDER='manual'.
handle-payment-webhookMaya / PayMongoMaya events are re-fetched with the secret key and reconciled by ID, status, reference, currency, and amount. PayMongo retains signature verification. A shared idempotent finalizer confirms the booking, inserts service details/add-ons/charges/vaccines/waivers/documents/payment, and sends email. Maya failure/cancel/expiry releases the pending booking.
get-payment-statusbooking.js (hosted return)Returns only pending/confirmed payment state by booking reference so a browser success redirect cannot bypass webhook confirmation.
submit-booking (646 L)Admin wizard · walk-in pageDirect full-booking creation without a gateway: owner upsert by email, pet insert, booking (status pending, source admin/walkin; admin then PATCHes status/payment), detail row per service, addons, charges, vaccines, waivers. Sends the confirmation email immediately when adminCreated + owner email.
cancel-pending-booking (58 L)booking.js (edit-after-failed-payment)Sets status='cancelled' with reason, only where status is still pending — can never cancel a confirmed booking.
get-upload-url (95 L)booking.js (vaccine files)Issues signed Supabase Storage upload URLs per file; resulting paths flow through create-payment metadata → webhook persists vaccine_documents rows.

Scheduled job

pg_cron (defined in the database): every 5 minutes, call expire_pending_bookings() to cancel pending/unpaid bookings whose provider-specific expires_at has passed. Manual transfers do not create pending holds; Maya uses one hour.

Supporting tables

TablePurpose
pricingSingle source of all rates (grooming/face-trim/addons/hotel/daycare/late/member %/convenience). Hydrates both frontends at runtime.
admin_usersEmail allow-list gating the admin app and authenticated RLS policies.
walkin_tokensSingle-use UUIDs minted by the admin FAB; verified + deleted by the walk-in gate.
blocked_schedulesresource_type (room/groomer/studio) + resource_id + dates[] + start/end time + reason + active.
resource_service_hoursDate-specific outer availability window per resource: service date, start/end time, last-service cutoff, and active flag. Grooming consumes it first.
booking_editsAudit rows written on admin edits (who + change blob).

Conventions

  • booking_date = creation date (matches created_at::date); the appointment date is service_date in the service-detail table (hotel: checkin_date/checkout_date). Every availability/check-in query keys on the latter.
  • Soft deletes via active flags (resources, blocks) and cancelled status (bookings); no operational hard deletes.
  • Detail-table embeds are promoted to !inner when the parent must be filtered by child columns (PostgREST).
  • Resource colors constrained to an 8-value CHECK list; they drive calendar rendering.
  • Grooming availability is the intersection of date-specific service hours, the last-service cutoff, full service duration, explicit blocked schedules, and existing assigned/unassigned bookings. No service-hours row means unavailable.

Security model

ConcernMechanism
Public (anon) accessThe anon key is shipped in client JS by design; RLS limits it to the reads the booking flow needs (branches, pricing, resources, availability projections) and the narrow writes it performs (e.g. consuming a walk-in token). No anon access to other customers' personal data.
Admin SSOGoogle OAuth via Supabase Auth → JWT. App-level gate + RLS policies both check membership in admin_users by JWT email (e.g. booking_charges: FOR ALL TO authenticated USING/WITH CHECK admin-exists).
Privileged writesOnly edge functions hold the service-role key; booking confirmation, detail/charge writes, and emails happen server-side. Maya webhooks are verified against Maya's Payments API; PayMongo retains signature verification.
IdempotencyWebhook double-fire (two event types per payment) is neutralized by conditional status transitions.
Walk-in gatingFee-free mode requires a single-use server-minted token; consumed atomically on entry.
Pending-hold abusepg_cron auto-cancel caps inventory lockup from abandoned checkouts at ~15 minutes.
Surface hygienerobots.txt disallows the legacy admin page and test suite; the React admin is unindexed and gate-protected; admin app sends x-client-info for traceability.
SecretsMaya, retained PayMongo, and Resend keys live only in Supabase function secrets; nothing secret is committed.

Build & deployment

  • Public site: zero build — edit, commit, push to main; GitHub Pages redeploys.
  • Admin: cd admin-src && npm run build → Vite emits hashed bundles into ../admin/, which are committed alongside the source change and served by Pages at /admin/.
  • Database: DDL changes belong in supabase/migrations/*.sql and are applied via the Supabase SQL editor/CLI. After DDL, reload PostgREST's schema cache (NOTIFY pgrst, 'reload schema';) — stale caches have caused phantom-relationship errors.
  • Edge functions: deployed with the Supabase CLI (supabase functions deploy <name>); env secrets managed in the dashboard.
  • Local preview: .claude/launch.json defines a static server (python http.server :8788) for the repo root.
A CI job ("chore: build admin app [skip ci]") also rebuilds and commits /admin/. It has once raced a source change and effectively reverted it; committed-bundle + bot-commit is the most fragile part of the pipeline (see §12 and the build retrospective).

Operational gotchas (learned the hard way)

GotchaDetail / standing fix
booking_date semanticsIt is the creation date. Filtering availability by it silently misses anything booked on a different day — always join the detail table's service_date.
Timezonescreated_at is UTC; PHT is UTC+8. Group/display using local conversions, never string-split the ISO timestamp.
Time formatsHotel/daycare hours are stored as bare hour strings ("18"); normalize (parseInt/toHHMM) before feeding time inputs or formatters — "6:undefined PM" class of bugs.
PostgREST embedsFiltering parents by child columns requires !inner; multiple FKs between two tables make embeds ambiguous (HTTP 300) — disambiguate table!fk_name(...); after DDL, NOTIFY pgrst, 'reload schema'.
RLS soft-failsAn INSERT-only policy on booking_charges let edits insert new lines while DELETEs silently failed → duplicate charges. Policy is now FOR ALL; treat silent catch-and-continue around DB writes as a smell.
Auth deadlockNever call supabase.auth.getSession() inside per-request header builders — it queues behind token refreshes. Use the cached-token pattern (lib/supabase.js).
React nested componentsComponents defined inside a component remount per render (input focus loss). Hoist to module scope or call as plain functions.
Committed bundles/admin/ hashed assets churn every build and conflict across branches/bots; long-term fix is building in CI (Pages artifact) instead of committing output.
